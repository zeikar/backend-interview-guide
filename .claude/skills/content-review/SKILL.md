# Content Review — 백엔드 면접 가이드 품질 검토

면접 가이드 콘텐츠의 기술적 정확성과 면접 적합성을 검토하는 스킬.

## 워크플로우

### Step 1: 기준 수립 (Dynamic Reference)

비교 기준 문서를 동적으로 선택한다:

1. 대상 문서의 카테고리 디렉토리에서 `Glob("{category}/*.md")`로 파일 목록 수집
2. README.md를 제외한 `.md` 파일의 줄 수를 확인하여 가장 긴 문서 2개를 기준 문서로 선택
3. 카테고리에 문서가 1개 이하이면, 다른 카테고리에서 가장 긴 문서 1개를 추가 참조
4. 기준 문서를 Read하여 구조, 깊이, 스타일의 비교 기준선 수립
5. `AGENTS.md`의 Writing Tips도 스타일 기준에 포함

### Step 2: 기술 정확성 검토

| 검토 항목 | 확인 내용 |
|----------|---------|
| 개념 정의 | 기술 용어의 정의가 정확한가? 오해를 유발하는 표현은 없는가? |
| 동작 원리 | 내부 동작 설명이 실제 구현과 일치하는가? |
| 비교 분석 | A vs B 비교가 공정하고 정확한가? 각각의 적합한 사용 사례를 포함하는가? |
| 코드 예시 | 코드가 문법적으로 올바르고 실행 가능한가? 안티패턴을 사용하지 않는가? |
| 수치/성능 | 성능 수치나 벤치마크 데이터가 합리적인가? |
| 최신성 | 더 이상 권장되지 않는 기술이나 방식을 소개하고 있지 않은가? |

### Step 3: 면접 적합성 평가

면접관 관점에서 다음을 평가한다:

1. **깊이** — "왜?"라는 후속 질문에 답할 수 있는 수준인가?
   - 부족: "Redis는 빠릅니다" (이유 없음)
   - 충분: "Redis는 인메모리 데이터 스토어로, 디스크 I/O를 회피하여 마이크로초 단위의 응답을 제공합니다"

2. **실무 연결** — 이론만이 아닌 실무 시나리오를 포함하는가?
   - 부족: CAP 정리의 정의만 나열
   - 충분: CAP 정리 설명 + "은행 시스템은 일관성을 우선, SNS는 가용성을 우선"

3. **트레이드오프** — 장점만이 아닌 단점과 한계도 다루는가?

4. **범위** — 신입/경력 모두에게 유용한 수준의 범위인가?

5. **면접관 시뮬레이션** — 각 주요 섹션에 대해 후속 질문 1개를 생각하고, 문서 내용만으로 답할 수 있는지 판단한다.

### Step 4: 스타일 일관성 확인

다음 기준으로 검토한다. 기준 문서와 비교하여 차이점을 식별한다.

| 항목 | 기준 |
|------|------|
| 제목 형식 | `# 한국어 (English)` 패턴. 제품명은 canonical form |
| 목차 | 문서 최상단, 앵커 링크 사용 |
| 구분선 | 주요 섹션 사이 `---` |
| 볼드 라벨 | `- **키워드:** 설명` 형식 |
| 비교표 | 첫 열이 비교 기준 |
| 코드 블록 | 언어 태그 필수 |
| 톤 | AGENTS.md Writing Tips 준수 (가이드 톤, canonical 제품명, 자연스러운 한국어) |

### Step 5: 등급 판정 (Calibrated Rubric)

오케스트레이터가 이 등급을 파싱하여 워크플로우를 제어한다. 엄격히 따른다.

#### 상 (Publish-Ready)

아래 **모든** 조건 충족:
- 기술적 오류 없음
- 개념 → 원리 → 트레이드오프 → 실무 시나리오 흐름 완비
- "왜?" 후속 질문에 답할 수 있는 깊이
- 기존 문서와 동일한 스타일
- AGENTS.md Writing Tips 위반 없음

#### 중 (Needs Polish)

아래 **하나 이상** 해당:
- 핵심 내용은 정확하지만 깊이 부족 섹션 1~3개
- 트레이드오프 일부 누락
- 스타일 불일치 3곳 이하
- 구체적 수정 목록으로 해결 가능

#### 하 (Needs Rewrite)

아래 **하나 이상** 해당:
- 기술적 오류로 면접에서 오답 판정될 내용
- 문서 구조가 기존 패턴과 크게 다름
- 핵심 개념 누락
- AGENTS.md 톤 전반적 위반

등급이 경계선상이면 낮은 쪽으로 판정한다.

### Step 6: 리뷰 보고서 작성

다음 형식을 엄격히 따른다. 오케스트레이터가 `<!-- REVIEW_SUMMARY -->` 블록을 파싱한다.

```markdown
## 리뷰 요약

- **대상 문서:** [파일 경로]
- **비교 기준 문서:** [기준 문서 1], [기준 문서 2]
- **전체 평가:** 상|중|하
- **기술 정확성:** 상|중|하
- **면접 적합성:** 상|중|하
- **스타일 일관성:** 상|중|하

<!-- REVIEW_SUMMARY
overall: 상|중|하
accuracy: 상|중|하
interview_fit: 상|중|하
style: 상|중|하
critical_count: N
-->

## 수정 필요 사항 (Critical)

수정하지 않으면 면접 가이드로서 부적합한 항목.

1. **[위치/섹션]** — [문제 설명]
   - 수정 제안: [구체적 수정 내용]
   - 이유: [왜 수정이 필요한지]

## 보강 추천 사항 (Enhancement)

있으면 좋지만 필수는 아닌 개선 항목.

1. **[추가할 내용]** — [이유]

## 잘된 부분 (Positive)

유지해야 할 좋은 패턴.

1. [잘된 부분 설명]
```

## 에러 핸들링

- 전문 지식 외 영역은 `[추가 검증 권장]`으로 표시한다
- 비교 기준 문서가 없으면 `references/style-guide.md`만으로 스타일 검토한다
- 대상 문서가 존재하지 않으면 리뷰를 중단하고 보고한다
