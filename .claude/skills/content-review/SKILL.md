# Content Review — 백엔드 면접 가이드 품질 검토

면접 가이드 콘텐츠의 기술적 정확성과 면접 적합성을 검토하는 스킬.

## 워크플로우

### Step 1: 기준 수립

리뷰 전 같은 카테고리의 우수 문서를 1~2개 읽어 기준을 수립한다.
- database 카테고리: `caching.md`, `scaling.md`
- cloud 카테고리: `microservices.md`, `kubernetes.md`
- programming 카테고리: `go.md`, `javascript.md`
- system-design 카테고리: 아직 콘텐츠 없으므로 다른 카테고리 참조

### Step 2: 기술 정확성 검토

| 검토 항목 | 확인 내용 |
|----------|---------|
| 개념 정의 | 기술 용어의 정의가 정확한가? 오해를 유발하는 표현은 없는가? |
| 동작 원리 | 내부 동작 설명이 실제 구현과 일치하는가? |
| 비교 분석 | A vs B 비교가 공정하고 정확한가? 각각의 적합한 사용 사례를 포함하는가? |
| 코드 예시 | 코드가 문법적으로 올바르고 실행 가능한가? 안티패턴을 사용하지 않는가? |
| 수치/성능 | 성능 수치나 벤치마크 데이터가 합리적인가? |
| 최신성 | 더 이상 권장되지 않는 기술이나 방식을 소개하고 있지 않은가? |

### Step 3: 면접 적합성 평가

면접관 관점에서 다음을 평가한다:

1. **깊이** — "왜?"라는 후속 질문에 답할 수 있는 수준인가?
   - 나쁜 예: "Redis는 빠릅니다" (이유 없음)
   - 좋은 예: "Redis는 인메모리 데이터 스토어로, 디스크 I/O를 회피하여 마이크로초 단위의 응답을 제공합니다"

2. **실무 연결** — 이론만이 아닌 실무 시나리오를 포함하는가?
   - 나쁜 예: CAP 정리의 정의만 나열
   - 좋은 예: CAP 정리 설명 + "은행 시스템은 일관성을 우선, SNS는 가용성을 우선"

3. **트레이드오프** — 장점만이 아닌 단점과 한계도 다루는가?

4. **범위** — 신입/경력 모두에게 유용한 수준의 범위인가?

### Step 4: 스타일 일관성 확인

| 항목 | 기준 |
|------|------|
| 제목 형식 | `# 한국어 (English)` 패턴 |
| 목차 | 문서 최상단, 앵커 링크 사용 |
| 구분선 | 주요 섹션 사이 `---` |
| 볼드 라벨 | `- **키워드:** 설명` 형식 |
| 비교표 | 첫 열이 비교 기준 |
| 코드 블록 | 언어 태그 필수 |

### Step 5: 리뷰 보고서 작성

다음 형식으로 리뷰 결과를 정리한다:

```markdown
## 리뷰 요약

- **대상 문서:** [파일 경로]
- **전체 평가:** 상/중/하
- **기술 정확성:** 상/중/하
- **면접 적합성:** 상/중/하
- **스타일 일관성:** 상/중/하

## 수정 필요 사항 (Critical)

수정하지 않으면 면접 가이드로서 부적합한 항목.

1. **[위치/섹션]** — [문제 설명]
   - 수정 제안: [구체적 수정 내용]
   - 이유: [왜 수정이 필요한지]

## 보강 추천 사항 (Enhancement)

있으면 좋지만 필수는 아닌 개선 항목.

1. **[추가할 내용]** — [이유]

## 잘된 부분 (Positive)

유지해야 할 좋은 패턴. 향후 문서 작성의 참고점.

1. [잘된 부분 설명]
```

## 에러 핸들링

- 전문 지식 외의 영역은 `[추가 검증 권장]`으로 표시한다
- 비교 대상이 되는 문서가 아직 없으면 독립적으로 검토한다
